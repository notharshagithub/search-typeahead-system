CHAPTER 1: INTRODUCTION & SYSTEM REQUIREMENTS
================================================================================
The Typeahead System (also known as Autocomplete or Search Suggestions) is a highly complex engineering challenge disguised as a simple UI feature. When a user navigates to a search bar and begins typing, the system must predict what the user is searching for by returning the highest-ranked queries matching the current prefix.

1.1 Functional Requirements
- As a user types, the system must return up to 10 suggestions matching the prefix.
- The system must rank these suggestions based on historical search popularity.
- When a user selects a suggestion or hits Enter, the system must record this search to update its popularity ranking.
- The system must display a real-time list of "Trending Topics".

1.2 Non-Functional Requirements (The Hard Part)
- Ultra-Low Latency: Suggestions must appear in under 50ms. Human perception notices lag beyond 100ms.
- High Throughput (Reads): A single word typed by a user generates multiple API requests (e.g., typing "docker" generates requests for "d", "do", "doc", "dock", "docke", "docker"). The system must handle immense read volume.
- High Throughput (Writes): When a major event occurs, millions of users might search for the same term simultaneously. The database must not crash under heavy write loads.
- High Availability: If a cache node crashes, the system must continue functioning without downtime.

================================================================================
CHAPTER 2: HIGH-LEVEL ARCHITECTURE OVERVIEW
================================================================================
To meet the stringent non-functional requirements, we bypassed monolithic design in favor of a Microservices Architecture running inside Docker containers.

2.1 The Components
1. The Client (React Frontend): Handles user input, debounces keystrokes to prevent network spam, and renders results.
2. The API Gateway/Backend (Node.js/Express): The traffic controller. It routes read requests to the cache, write requests to the batching queue, and trending requests to the in-memory window.
3. The Database (SQLite): The persistent disk storage. It holds the canonical truth of all queries and their counts.
4. The Cache Cluster (3x Node.js Express Nodes): Ephemeral in-memory key-value stores. They hold recent read results to prevent repetitive database queries.

2.2 The Flow of a Read Request
1. User types "mac". Frontend waits 300ms, then sends GET /suggest?q=mac.
2. Backend receives "mac". It hashes the word "mac" to figure out which cache node owns it.
3. Backend asks Cache Node 2. If it has it, it returns instantly.
4. If Cache Node 2 misses, Backend asks SQLite.
5. SQLite traverses its B-Tree index, finds 10 results, and returns them to the Backend.
6. Backend returns results to Frontend, and asynchronously fires a request to Cache Node 2 to save the results for 60 seconds.

CHAPTER 3: FRONTEND ARCHITECTURE & REACT DEEP DIVE
================================================================================
The frontend is built using React (bootstrapped via Vite for lightning-fast HMR and bundling). 

3.1 Debouncing (The First Line of Defense)
We cannot send an API request for every single keystroke. We built a custom `useDebounce` React Hook.

CODE IMPLEMENTATION:
```javascript
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);
  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);
    return () => {
      clearTimeout(handler);
    };
  }, [value, delay]);
  return debouncedValue;
}
```
HOW IT WORKS: 
When a user types "i", `value` changes. `useEffect` triggers and starts a `setTimeout` for 300ms. If the user types "p" after 100ms, `value` changes again. Before running the new `useEffect`, React runs the cleanup function of the previous `useEffect`, which calls `clearTimeout(handler)`. The first timer is destroyed. A new 300ms timer starts. When the user finally stops typing for 300ms, `setDebouncedValue` fires. The main `App.jsx` component listens to `debouncedValue` and fires the `fetch()` request. This reduces network load by ~80%.

3.2 User Interface & UX
- Glassmorphism: We used CSS `backdrop-filter: blur(10px)` with semi-transparent `rgba(25, 25, 35, 0.7)` backgrounds to create a modern, sleek, "glass" aesthetic.
- Keyboard Navigation: We used the `onKeyDown` synthetic event in React. If `e.key === 'ArrowDown'`, we increment an active index state. If `e.key === 'Enter'`, we trigger the search.
- Text Highlighting: We use a utility function that splits the suggestion string based on the user's input prefix and wraps the matching part in a `<strong>` tag to make it bold.

CHAPTER 4: BACKEND API DESIGN & EXPRESS ORCHESTRATION
================================================================================
The backend is a Node.js server using Express.

4.1 Middleware Setup
- `cors()`: Cross-Origin Resource Sharing. Browsers block requests from `localhost:5173` to `localhost:3001` for security. CORS adds HTTP headers (`Access-Control-Allow-Origin`) to permit the frontend to communicate.
- `express.json()`: Intercepts incoming HTTP requests with `Content-Type: application/json`, parses the raw byte stream into a JavaScript Object, and attaches it to `req.body`.
- `morgan('dev')`: A logging middleware that prints every incoming HTTP request method, path, status code, and response time to the console.

4.2 Route Modularity
Instead of putting all endpoints in `server.js`, we used `express.Router()`:
- `routes/suggest.js`: Handles reads.
- `routes/search.js`: Handles writes.
- `routes/trending.js`: Handles trending topics.
- `routes/analytics.js`: Handles system health.
- `routes/cacheDebug.js`: Handles cache invalidation.

CHAPTER 5: DATABASE LAYER IN EXTREME DEPTH (SQLITE, B-TREES, WAL)
================================================================================
We used `better-sqlite3`, a highly optimized, synchronous C++ binding for SQLite.

5.1 Why SQLite?
For a local Typeahead system, SQLite is perfect because it stores data in a single file on the host operating system, avoiding the network overhead of connecting to an external database like PostgreSQL.

5.2 B-Tree Indexing (The Secret to Fast Reads)
We execute: `CREATE INDEX IF NOT EXISTS idx_query_prefix ON queries(query);`
Under the hood, SQLite creates a Balanced Tree (B-Tree) data structure.
- Without an Index (Full Table Scan): To find words starting with "mac", SQLite must read all 130,000 rows. Time Complexity: O(N). Extremely slow.
- With a B-Tree Index: SQLite traverses the tree nodes logarithmically to find the exact memory pointer where words starting with "mac" begin. It reads sequentially until the words stop starting with "mac", then stops. Time Complexity: O(log N). Blazing fast.

5.3 WAL Mode (Write-Ahead Logging)
We execute: `db.pragma('journal_mode = WAL');`
- Standard Rollback Journal: When writing, SQLite locks the entire database. Readers are blocked. If thousands of users are searching, a single write causes a traffic jam.
- WAL Mode: Writes are appended to a separate file (the WAL file). Readers continue reading from the original database file undisturbed. Periodically, SQLite "checkpoints" the WAL file into the main database. This enables immense concurrency.

CHAPTER 6: DISTRIBUTED CACHE & CONSISTENT HASHING MATHEMATICS
================================================================================
To prevent the SQLite database from being hammered, we built 3 in-memory Cache Nodes. But how do we decide which node stores which data?

6.1 The Flaw of Modulo Hashing
Naive approach: `hash("apple") % 3 nodes = Node 1`. 
If Node 1 crashes, the formula becomes `hash("apple") % 2 nodes = Node 0`. Suddenly, ALL keys point to the wrong nodes. 100% Cache Miss rate! The database gets destroyed by a stampede.

6.2 Consistent Hashing implementation
We built a custom Hash Ring in `backend/src/cache/hashRing.js` using Node's `crypto` module.

1. VIRTUAL NODES: We create 150 "virtual" replicas of each of the 3 physical nodes (450 total points). This ensures data is distributed evenly around the ring, preventing "hotspots".
2. HASHING THE NODES: 
   ```javascript
   const hashStr = crypto.createHash('md5').update('cache-node-1-vnode-45').digest('hex');
   const integerPos = parseInt(hashStr.substring(0, 8), 16);
   ```
   We convert the MD5 string into a 32-bit Integer and place it on our ring (an Array).
3. HASHING THE QUERY: When a user searches "apple", we hash "apple" using the exact same MD5 logic to get an Integer (e.g., 5000).
4. BINARY SEARCH (`bisectLeft`): We need to find the first Virtual Node whose Integer is greater than 5000. 
   We implemented a custom Binary Search loop. It cuts the 450-item array in half repeatedly, finding the target in just 8 steps (O(log N)).
5. We look up which physical node owns that Virtual Node, and route the HTTP request there.
If a node crashes, we remove its 150 virtual nodes from the ring. Only the keys immediately adjacent to it are remapped. 66% of the cache remains perfectly intact.

CHAPTER 7: THE READ-THROUGH CACHE IMPLEMENTATION FLOW
================================================================================
The Cache Nodes themselves are simple Express servers storing data in a native JavaScript `Map`. 

7.1 The Flow in `suggest.js`:
1. Check the Hash Ring to find the target node (`targetNode = defaultRing.getNodeForKey(prefix)`).
2. Fetch from cache.
   - We use `fetch(\`http://${targetNode}:3002/cache/${prefix}\`, { signal: AbortSignal.timeout(100) })`.
   - The `AbortSignal.timeout(100)` is critical. If the cache node is dead and hanging, the request aborts after 100ms. We instantly catch the error and fall back to SQLite. The user doesn't notice the failure.
3. Database Fallback: If cache misses, we query SQLite (`queryStore.getByPrefix`).
4. Asynchronous Cache Population:
   - We call `cacheClient.set()`. We DO NOT `await` this call.
   - We append `.catch(err => ...)` to silently log errors.
   - We immediately return the HTTP response to the user. The cache is populated in the background asynchronously (Fire-and-Forget). This saves precious milliseconds on the user's response time.

CHAPTER 8: WRITE-BATCHING & SOLVING THE DATABASE STAMPEDE
================================================================================
When a user hits Enter, we call `POST /search`. If 10,000 users hit Enter during the Super Bowl, 10,000 DB writes will bottleneck the disk I/O.

8.1 In-Memory Buffering (`batchQueue.js`)
We intercept the writes in RAM.
```javascript
const currentBatch = new Map();
function recordSearch(query) {
  currentBatch.set(query, (currentBatch.get(query) || 0) + 1);
}
```
If 10,000 users search "superbowl", the database isn't touched. The Map simply updates: `{"superbowl": 10000}`.

8.2 The Flush Interval
We run `setInterval(flushBatch, 5000)`. Every 5 seconds:
1. We check if `currentBatch` has data.
2. We assign `const batchToProcess = currentBatch;` and create a brand new `currentBatch = new Map();` for incoming traffic.
3. We convert the Map into an Array of objects: `[{query: "superbowl", count: 10000}]`.
4. We pass it to `queryStore.bulkUpsert()`.

8.3 SQLite Bulk Upsert
In `queryStore.js`, we wrap the array in a database `transaction()`. 
```sql
INSERT INTO queries (query, count, last_searched_at, recent_count)
VALUES (@query, @count, @now, @count)
ON CONFLICT(query) DO UPDATE SET 
count = count + excluded.count,
recent_count = recent_count + excluded.count,
last_searched_at = @now;
```
This query tells SQLite: "Insert the word. If it already exists (ON CONFLICT), just add the new count to the existing count." Doing this in a single transaction takes <10ms for thousands of queries. We turned 10,000 disk writes into 1.

CHAPTER 9: REAL-TIME TRENDING VIA IN-MEMORY SLIDING WINDOWS
================================================================================
How do we know what's trending without running heavy `ORDER BY count DESC` queries on the database? We leverage the buffers from Chapter 8!

9.1 Sliding Window Algorithm
In `batchQueue.js`, we have an Array: `const slidingWindow = []`.
Every 5 seconds, when `flushBatch()` runs, we push the flushed `Map` into the `slidingWindow` array.
To prevent infinite memory growth, we enforce a strict size:
`if (slidingWindow.length > 10) { slidingWindow.shift(); }`
`shift()` removes the oldest item from the front of the array. The array now represents exactly the last 50 seconds of real-time search activity.

9.2 Serving the Trending API
When `GET /trending` is called:
1. We create an empty `Map`.
2. We loop through all 10 Maps inside `slidingWindow`, adding up the counts for every word.
3. We convert the final aggregated Map into an Array.
4. We run `Array.prototype.sort((a,b) => b.count - a.count)`.
5. We run `.slice(0, 10)` to get the top 10.
Because this operates entirely in CPU L1/L2 Cache (RAM), it takes <1ms and puts ZERO load on the database.

CHAPTER 10: OBSERVABILITY, METRICS & ANALYTICS
================================================================================
To monitor system health, we built `store/metricsStore.js`.
- It tracks `cacheHits`, `cacheMisses`, and `totalSearches`.
- It keeps a sliding window array of the last 100 API response latencies.
- When `GET /analytics` is called, it calculates `(cacheHits / total) * 100` to yield the `hitRatePercent`, and averages the latency array using `Array.prototype.reduce`.
- The Frontend `AnalyticsPanel` polls this endpoint every 2 seconds, providing a live command center. As users hit the same queries, they literally watch the Cache Hit Rate climb and the Average Latency drop to <5ms.

CHAPTER 11: DATA INGESTION & IDEMPOTENT BOOTSTRAPPING
================================================================================
The system parses a massive 130,193-row JSON dataset upon booting.
- Idempotency: `queryStore.initDB()` runs a `SELECT count(*) FROM queries`. If it equals 0, it calls `loadDataset()`. This ensures we don't accidentally load the JSON file twice if the server restarts.
- `fs.readFileSync`: It reads the 10MB JSON file into memory, parses it with `JSON.parse()`, and feeds it directly into the `bulkUpsert` transaction logic outlined in Chapter 8. It ingests all 130k rows in exactly ~440ms.

CHAPTER 12: DOCKER INFRASTRUCTURE, NETWORKING & HEALTHCHECKS
================================================================================
The entire system is orchestrated by `docker-compose.yml`.
- `networks`: A custom `bridge` network isolates the containers from the host machine.
- Internal DNS: The backend doesn't use hardcoded IP addresses. It `fetch`es `http://cache-node-1:3002`. Docker's embedded DNS server dynamically resolves the hostname `cache-node-1` to the correct internal container IP address.
- `depends_on` & `healthcheck`: The backend requires the cache nodes to be ready before it boots. Each cache node has a `healthcheck` that runs `wget --spider http://localhost:3002/health`. Docker pauses the backend boot sequence until the cache nodes report a `Healthy` status.
- `volumes`: The `/app/data` directory inside the backend container is mapped to `./data` on the host machine. If we destroy the docker containers, the SQLite `.db` file remains safe on the physical hard drive.

CHAPTER 13: CONCLUSION & TIME COMPLEXITY SUMMARY
================================================================================
By utilizing specific Data Structures and Architectures, we successfully manipulated the Time Complexity of our system to achieve massive scale:

1. B-Tree Indexes: Reduced DB Prefix Searches from O(N) to O(log N).
2. Hash Ring (Binary Search): Reduced cache routing lookups from O(N) to O(log N).
3. JavaScript Maps (Hash Tables): Reduced Write Batching lookups and increments to O(1) constant time.
4. Asynchronous Queues: Reduced Disk I/O bottlenecks to O(1) event-loop offloading.
5. Sliding Windows: Reduced real-time trending queries from O(DB_SIZE log DB_SIZE) down to O(K log K) where K is strictly the number of unique terms searched in the last 50 seconds.

This system is a robust, fault-tolerant, horizontally scalable Typeahead engine ready for production workloads.
