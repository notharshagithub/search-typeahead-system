CHAPTER 1: INTRODUCTION & SYSTEM GOALS
================================================================================
The primary goal of this project was to design and implement a highly scalable, Google-style "Typeahead" (Autocomplete) Search System. 

When a user types into a search box, the system must instantly return the most popular search queries that match the prefix. Because a single user typing a word generates multiple API requests, the system handles heavy read traffic and must be optimized for ultra-low latency. Furthermore, when users finalize a search, the system must record it to update the popularity rankings without overloading the database.

================================================================================
CHAPTER 2: SYSTEM ARCHITECTURE OVERVIEW
================================================================================
The system is built using a Microservices Architecture pattern. The components are decoupled, allowing them to scale independently.

* Frontend: React.js Single Page Application (SPA).
* Backend: Node.js / Express.js REST API.
* Database: SQLite (acting as the persistent disk store).
* Distributed Cache: A custom-built, multi-node, in-memory caching cluster.
* Hash Ring: A Consistent Hashing mechanism to distribute data evenly across cache nodes.

================================================================================
CHAPTER 3: INFRASTRUCTURE & DOCKERIZATION
================================================================================
The entire application runs inside isolated Docker containers, orchestrated by Docker Compose. This ensures a "write once, run anywhere" environment.

* Service Definitions:
  - `frontend`: Exposes port 5173.
  - `backend`: Exposes port 3001.
  - `cache-node-1`, `cache-node-2`, `cache-node-3`: Expose port 3002 internally.
* Networking: All containers communicate securely over a custom bridge network named `typeahead-net`.
* Healthchecks: Every container possesses a built-in healthcheck (`wget --spider`) to ensure services boot in the correct dependency order. The backend waits for the cache nodes; the frontend waits for the backend.
* Volumes: The `data/` and `dataset/` directories are mounted via volumes to persist the SQLite database file across container restarts.

================================================================================
CHAPTER 4: FRONTEND IMPLEMENTATION (REACT, UI, UX)
================================================================================
The frontend is designed to be highly responsive and aesthetically pleasing, using a dark-mode "glassmorphism" design.

* Technologies: React, Vite, standard CSS.
* Keyboard Navigation: Users can use the Up and Down arrow keys to navigate the suggestion dropdown, and hit Enter to select a query. The `/` key instantly focuses the search bar.
* Debouncing: A critical frontend optimization. We implemented a custom `useDebounce` hook set to 300ms. The UI waits until the user stops typing for 300 milliseconds before firing a network request. This reduces network spam by over 80%.
* Visual Highlighting: The dropdown dynamically boldens the matching prefix in the suggestions (e.g., typing "ip" makes "ip" bold in "**ip**hone").
* Animations: CSS keyframe animations provide smooth fade-ins and scale effects.

================================================================================
CHAPTER 5: BACKEND API LAYER
================================================================================
The Node.js/Express backend serves as the traffic controller. It is heavily modularized into different router files.

* `GET /suggest?q=<prefix>`: The core endpoint. It checks the cache layer first, falls back to the database on a miss, asynchronously populates the cache, and returns the top 10 suggestions.
* `POST /search`: The write endpoint. Called when a user hits Enter. It immediately returns `200 OK` and pushes the write event into an asynchronous queue.
* `GET /trending`: Returns the top trending searches of the last minute.
* `GET /analytics`: Exposes internal system metrics.
* `GET /cache/debug` & `POST /cache/flush`: Administrative routes for managing the cache cluster.

================================================================================
CHAPTER 6: DATABASE LAYER (SQLITE, WAL, INDEXES)
================================================================================
SQLite acts as the persistent, disk-based source of truth.

* Schema: The `queries` table holds `query` (Primary Key), `count` (popularity), `last_searched_at`, and `recent_count`.
* B-Tree Indexing: An index was created on the `query` column. This changes the Time Complexity of a prefix search from O(N) full-table scan to O(log N) tree traversal.
* WAL Mode: Write-Ahead Logging is enabled. This alters how SQLite writes to disk, allowing concurrent reads and writes. A reader does not block a writer, which is vital for high throughput.

================================================================================
CHAPTER 7: DATA INGESTION & SEEDING
================================================================================
The system must be able to boot up and ingest massive datasets instantly.

* Initially, a synthetic dataset of 110,000 queries was generated using a custom Node script.
* Later, the system was upgraded to ingest a real-world JSON dataset of 130,193 queries.
* Idempotent Boot: Upon startup, the `queryStore` checks if the database is empty. If it is, it reads the JSON dataset from the disk.
* Bulk Upsert: Instead of firing 130,000 `INSERT` commands, the system wraps all entries into a single database `transaction`. It uses the `ON CONFLICT DO UPDATE` clause. This allows the system to parse and ingest 130,000 records into the database in roughly 400 milliseconds.

================================================================================
CHAPTER 8: DISTRIBUTED CACHING STRATEGY
================================================================================
To shield the database from massive read traffic, an ephemeral caching layer was built.

* Independent Nodes: Three separate Node.js servers run barebones Express servers acting purely as Key-Value stores using JavaScript `Map` objects.
* Consistent Hashing: The backend uses a `HashRing` to map prefixes to cache nodes. Using MD5 hashing and binary search, it distributes the load evenly.
* Virtual Nodes: To prevent "hotspots" (where one node gets all the traffic), the Hash Ring assigns 150 "virtual nodes" to each physical node.
* Read-Through Pattern: The backend orchestrates the logic. It asks the Cache Node. If missing, it asks the DB. It then asynchronously (`fire-and-forget`) sends the data to the Cache Node.
* TTL & Eviction: Every cache entry deletes itself after 60 seconds to ensure stale data is rotated out.
* Cache Invalidation API: We implemented specific endpoints that can traverse the Hash Ring, find the exact node holding a stale key, and send a `DELETE` request to forcibly clear it.

================================================================================
CHAPTER 9: WRITE BATCHING & DATABASE PROTECTION
================================================================================
Protecting the database from a stampede of `POST /search` write events.

* Problem: 1,000 users searching for "docker" simultaneously triggers 1,000 DB writes.
* Solution: In-Memory Write Batching. 
* Implementation: When `POST /search` is called, the query is pushed to an in-memory Map `{"docker": 1}`. Subsequent requests increment this in-memory counter `{"docker": 1000}`.
* Every 5000ms (5 seconds), a `setInterval` worker takes the entire Map, converts it to an array, and writes it to SQLite in one single `bulkUpsert` transaction.
* Result: 1,000 individual write operations are condensed into exactly 1 write operation.

================================================================================
CHAPTER 10: REAL-TIME TRENDING TOPICS
================================================================================
Providing users with a "Trending Now" discovery feature.

* Since Chapter 9 already buffers searches every 5 seconds, we leverage this existing memory buffer.
* The `batchQueue.js` maintains a "Sliding Window" array containing the last 10 buffers (representing exactly 50 seconds of real-time search activity).
* The `GET /trending` API iterates over this in-memory window, sums up the counts, sorts them, and returns the top 10.
* Because this is entirely CPU-bound in memory, it requires ZERO database queries to compute trending topics.

================================================================================
CHAPTER 11: PERFORMANCE MONITORING & ANALYTICS
================================================================================
An observability layer to monitor system health without external dependencies.

* Metrics Store: An in-memory singleton object tracking `cacheHits`, `cacheMisses`, `totalSearches`, and a 100-item array of API response times (`latencies`).
* Instrumentation: The `/suggest` route explicitly records whether a request was served by Cache or DB, and logs the execution time.
* Dashboard: The React frontend features an Analytics Panel that polls the `GET /analytics` endpoint every 2 seconds.
* It displays Uptime, Cache Hit Rate (%), Average Latency (ms), and Total Requests. When users type identical queries, they visually witness the latency drop from ~35ms to ~2ms as the Cache Hit Rate spikes.

================================================================================
CHAPTER 12: FAULT TOLERANCE & EDGE CASES ADDRESSED
================================================================================
* Cache Node Failures: If a cache node crashes, the backend HTTP fetch requests have a strict `AbortSignal.timeout(100)` of 100ms. If the cache node doesn't respond in 100ms, the backend immediately fails over to the SQLite database. The user never notices the crash.
* Empty Queries: The backend explicitly drops empty spaces and empty queries before they hit the database.
* Case Insensitivity: Both the backend formatting `.toLowerCase()` and SQLite's `LIKE` clause handle mixed-case inputs gracefully.
* Asynchronous Non-Blocking Cache Sets: When populating the cache after a database hit, the backend uses `.catch(err => ...)` without an `await`. It returns the HTTP response to the user immediately, while populating the cache node in the background.
